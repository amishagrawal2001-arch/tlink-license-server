Tlink License Server

A centralized license management server for Tlink products. Generate, activate, validate, and manage license keys for your Electron/desktop applications with a full-featured admin dashboard and REST API.

ⓘ
Base URL: http://localhost:4000 (development)
All API endpoints are prefixed with /api.

Quick Start

  1. Clone and install dependencies
    bash
    git clone <repo-url> tlink-license-server
cd tlink-license-server
npm install
  2. Configure environment
    bash
    cp .env.example .env
# Edit .env with your settings (see Configuration section)
  3. Seed the database
    bash
    npm run seed

    This creates an admin user, registers the NetOps app (NO), and generates sample license keys.

  4. Start the server
    bash
    # Development (with auto-reload)
npm run dev

# Production
npm start
  5. Open the admin dashboard

    Navigate to http://localhost:4000/admin and login with admin / admin123.

Database Setup

The server uses a flat-file JSON database stored at the configured DATABASE_PATH (default: data/licenses.json). No external database is required.

The database is automatically created on first run. It contains four collections:

CollectionPurpose
appsRegistered applications (app_code, app_name)
license_keysGenerated license keys with metadata
activationsMachine activations linked to keys
admin_usersAdmin users with bcrypt-hashed passwords
ⓘ
Run npm run seed after initial setup to populate the database with an admin user, the NetOps app, and sample license keys.

CORS Configuration

Set the CORS_ORIGINS environment variable to control which origins can make requests to the API.

  • * — Allow all origins (development only)
  • Comma-separated list — e.g., http://localhost:4200,app://tlink-netops

For Electron apps, use the app:// protocol scheme as your origin.

Rate Limiting

Two rate limit tiers are applied via express-rate-limit:

TierRoutesWindowMax Requests
Public/api/auth/*, /api/license/*60 seconds30 requests
Admin/api/keys/*, /api/dashboard/*60 seconds100 requests

Rate limit headers are returned in responses (RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset).

Authentication

Admin Login Flow

  1. POST credentials to /api/auth/login with username and password.
  2. Receive a JWT token in the response along with the user object.
  3. Include the token in subsequent requests as Authorization: Bearer <token>.
  4. Token expires in 24 hours. Re-authenticate to obtain a fresh token.
bash
# Login and get token
curl -X POST http://localhost:4000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "admin123"}'
json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "user": {
    "id": 1,
    "username": "admin",
    "role": "admin"
  }
}

JWT Token Usage

The JWT payload includes id, username, and role. Tokens are signed with the JWT_SECRET and expire after 24 hours.

Include the token in the Authorization header for all admin endpoints:

bash
curl http://localhost:4000/api/keys \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..."

Key Format

Tlink license keys follow a structured format that encodes application, tier, and expiry information with a checksum for integrity verification.

Structure

TLINK-NO-P4KR-2712-X9HJ-AB3F | | | | | | | | | | | +-- Checksum (FNV-1a hash, 4 chars) | | | | +---------- Random segment (4 chars) | | | +------------------ Expiry: YY=27, MM=12 (Dec 2027) | | +-------------------------- Tier char + 3 random (P=Pro, E=Enterprise) | +-------------------------------- App code (2 chars, padded with X) +---------------------------------------- Prefix (always TLINK)

Segment Details

SegmentLengthDescription
TLINK5Fixed prefix identifying Tlink keys
App Code2Two-letter application code, padded with X if needed
Tier Segment4First character encodes tier (P=Pro, E=Enterprise), followed by 3 random characters
Expiry4Two-digit year + two-digit month (YYMM)
Random4Random characters from charset A-Z (no I, O) + 2-9
Checksum4FNV-1a hash of all preceding segments + salt, encoded as 4 characters

Checksum Algorithm

The checksum is computed using FNV-1a (Fowler-Noll-Vo) hash:

  1. Concatenate all segments with dashes: TLINK-NO-P4KR-2712-X9HJ
  2. Append the KEY_SALT environment variable: TLINK-NO-P4KR-2712-X9HJ-<salt>
  3. Compute FNV-1a hash (32-bit)
  4. Convert to 4-character string using the charset: ABCDEFGHJKLMNPQRSTUVWXYZ23456789
ⓘ
The charset intentionally excludes I, O, 0, and 1 to avoid visual ambiguity when users type keys manually.

CharacterTierDescription
PProProfessional tier
EEnterpriseEnterprise tier

Client Integration

Guide for integrating Tlink license validation into Electron/Angular desktop applications.

License Activation on First Launch

typescript
interface LicenseResponse {
  valid: boolean;
  tier?: 'pro' | 'enterprise';
  expiry?: string;
  customer?: string;
  message?: string;
}

async function activateLicense(key: string): Promise<LicenseResponse> {
  const machineId = getMachineFingerprint(); // Your machine ID logic

  const response = await fetch('http://your-server:4000/api/license/activate', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      key,
      machineId,
      appCode: 'NO',
      appVersion: '1.0.0',
      os: process.platform
    })
  });

  return response.json();
}

Validate on Startup

typescript
async function validateOnStartup(savedKey: string): Promise<boolean> {
  try {
    const response = await fetch('http://your-server:4000/api/license/validate', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        key: savedKey,
        machineId: getMachineFingerprint()
      })
    });

    const data = await response.json();

    if (data.valid) {
      // Store tier info for feature gating
      localStorage.setItem('license_tier', data.tier);
      localStorage.setItem('license_expiry', data.expiry);
      return true;
    }

    // License invalid - show activation screen
    return false;
  } catch (err) {
    // Network error - use offline fallback
    return checkOfflineLicense(savedKey);
  }
}

Heartbeat Setup

typescript
const HEARTBEAT_INTERVAL = 4 * 60 * 60 * 1000; // 4 hours

function startHeartbeat(key: string) {
  const machineId = getMachineFingerprint();

  setInterval(async () => {
    try {
      const res = await fetch('http://your-server:4000/api/license/heartbeat', {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({ key, machineId })
      });
      const data = await res.json();

      if (!data.valid) {
        // License expired or revoked
        showLicenseExpiredDialog();
      }
    } catch (err) {
      // Server unreachable - continue with grace period
      console.warn('Heartbeat failed, will retry next interval');
    }
  }, HEARTBEAT_INTERVAL);
}

Offline Fallback

When the license server is unreachable, implement a grace period using locally cached license data:

typescript
function checkOfflineLicense(key: string): boolean {
  const lastValidated = localStorage.getItem('last_validated_at');
  const expiry = localStorage.getItem('license_expiry');
  const GRACE_DAYS = 7;

  if (!lastValidated || !expiry) return false;

  const daysSinceValidation = (Date.now() - new Date(lastValidated).getTime())
    / (1000 * 60 * 60 * 24);
  const notExpired = new Date() < new Date(expiry);

  return notExpired && daysSinceValidation < GRACE_DAYS;
}

Security Checklist

  • Change JWT_SECRET to a cryptographically random string (64+ chars)
  • Change KEY_SALT to a unique random string
  • Set a strong ADMIN_PASSWORD
  • Restrict CORS_ORIGINS to specific domains (no wildcard)
  • Use HTTPS with a reverse proxy (nginx, Caddy) in front of the server
  • Place behind a firewall; only expose port 443
  • Mount /data as a persistent volume for database durability
  • Set up regular backups of the data/licenses.json file
  • Monitor rate-limit headers for abuse patterns
  • Rotate JWT_SECRET periodically (forces re-authentication)

Backup Strategy

The database is a single JSON file. Back it up with a simple cron job:

bash
# Add to crontab (daily at 2 AM)
0 2 * * * cp /app/data/licenses.json /backups/licenses-$(date +\%Y\%m\%d).json

Troubleshooting

400: Invalid license key

The key format or checksum does not match. Common causes:

  • Key was typed incorrectly (check for O vs 0, I vs 1 confusion — these characters are excluded from valid keys)
  • The KEY_SALT on the server differs from when the key was generated. Keys generated with a different salt will not validate.
  • Key string was truncated during copy-paste
403: Max activations reached

The license key has been activated on the maximum number of machines. Solutions:

  • Deactivate a machine using POST /api/license/deactivate
  • Ask the admin to increase maxMachines via PUT /api/keys/:id
401: Missing or invalid authorization header

Admin API calls require the Authorization header:

  • Ensure the header format is exactly: Authorization: Bearer <token> (note the space after "Bearer")
  • Token may have expired (24-hour validity). Re-authenticate via POST /api/auth/login.
CORS errors in browser/Electron

The browser is blocking requests due to cross-origin policy:

  • Check that CORS_ORIGINS includes your app's origin
  • For Electron apps, the origin is typically app://<app-name> or file://
  • Set CORS_ORIGINS=* temporarily to confirm CORS is the issue
  • Ensure no typos in the origin URL (protocol, port, trailing slash)
CSP (Content Security Policy) blocking requests

If your Electron app has CSP enabled, it may block connections to the license server:

  • Add the license server URL to the connect-src directive in your CSP meta tag
  • Example: connect-src 'self' http://localhost:4000 https://license.yourdomain.com
  • Check the browser/Electron dev console for CSP violation messages
Server unreachable / ECONNREFUSED

The license server is not responding:

  • Verify the server is running: curl http://localhost:4000/api/health
  • Check the configured port matches your client URL
  • Ensure firewall rules allow traffic on port 4000
  • In Docker: check container logs with docker logs tlink-license
  • With PM2: check logs with pm2 logs tlink-license
Key validation passes but activation fails

The key structure is valid but the database record check fails:

  • The key may not exist in the database (was generated but never inserted)
  • The key status may be revoked
  • The appCode in the request may not match the key's associated app
Rate limit exceeded (429)

Too many requests from the same IP address. The server enforces:

  • Public routes: 30 requests per minute
  • Admin routes: 100 requests per minute
  • Check RateLimit-Remaining header to monitor usage
  • Implement exponential backoff in your client
