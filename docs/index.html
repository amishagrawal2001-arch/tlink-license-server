Tlink License Server

Centralized license management for Tlink products. End users sign in with email + password, the client sends a device fingerprint, and the server issues JWT access and refresh tokens scoped to a specific entitlement. Admins manage users, their entitlements, and per-device activations from the built-in dashboard.

ⓘ
Base URL: http://localhost:4000 (development)
All public license endpoints are mounted under /api/licenses/*. The legacy path /api/license/* remains as an alias for older clients.

How It Works

A user signs in once with email and password; the server mints access + refresh JWTs bound to the device's fingerprint. The app periodically heartbeats to prove it's still live, and refreshes tokens automatically. Offline grace lets the app keep working without network for a bounded window.

End-to-End License Lifecycle

Provisioning (Admin)

Admin opens /admin → Creates user (email + password) → Adds entitlement (product, type, max_devices)

First Activation (Client)

User enters email + password → POST /api/licenses/activate + fingerprint → Server auth + entitlement + device checks → Returns VALID envelope + access/refresh tokens

Access-Token Refresh (~every 13 min)

Client timer fires near expiry → POST /api/licenses/refresh (refresh_token + fp) → Fresh access + refresh pair

Heartbeat (every 4 hours)

POST /api/licenses/heartbeat { device_id, license_id } → Server bumps last_seen_at → { status: "VALID", reason_code: null }

Revocation / Expiry / Device Swap

Admin revokes seat or force-deactivates device → Next heartbeat returns SEAT_REVOKED / DEVICE_MISMATCH → Client clears keychain, re-shows sign-in

Sign-Out

POST /api/licenses/deactivate (Bearer + fp) → Activation marked deactivated (seat freed) → Client clears tokens, back to sign-in

Architecture

The server is a thin Express.js app backed by a single JSON file. End-user accounts, entitlements, activations, apps, and admin credentials are all stored in data/licenses.json. Entitlements live inside the user record — one row per account with an embedded array of product licenses.

Data Model

CollectionPurpose
usersEnd-user accounts. Each holds an embedded entitlements[] array, one per product.
appsRegistered products. Identified by app_code. Optional blocked_app_versions[].
activationsOne row per (user × license × device). Stores fingerprint, platform, app version, last_seen_at, deactivated_at.
admin_usersAdmin dashboard logins. Separate pool from end users; tokens carry typ: "admin".

Database Setup

The server uses a flat-file JSON database stored at the configured DATABASE_PATH (default: data/licenses.json). No external database is required.

The database is automatically created on first run. It contains four collections:

CollectionPurpose
appsRegistered applications (app_code, app_name, optional blocked_app_versions[])
usersEnd-user accounts, each with embedded entitlements[] per product
activationsOne row per (user × entitlement × device fingerprint); tracks last_seen_at, deactivated_at, activation_source
admin_usersAdmin dashboard logins, bcrypt-hashed
ⓘ
Run npm run seed after initial setup to create the default admin user, register product apps, and populate a few sample user accounts with entitlements.

CORS Configuration

Set the CORS_ORIGINS environment variable to control which origins can make requests to the API.

  • * — Allow all origins (development only)
  • Comma-separated list — e.g., http://localhost:4200,app://tlink-netops

For Electron apps, use the app:// protocol scheme as your origin.

Authentication

Two completely separate token worlds live on the same server, signed with the same JWT_SECRET but distinguished by a typ claim. End-user license tokens cannot access admin endpoints.

Token typetyp claimTTLIssued byAccepted by
Adminadmin24hPOST /api/auth/login/api/users, /api/apps, /api/dashboard, /api/settings
Accessaccess15 minPOST /api/licenses/activate and /refresh/api/licenses/validate, /heartbeat, /deactivate
Refreshrefresh7 daysSame as access/api/licenses/refresh — and inline via /heartbeat when access is expired

Admin Login Flow

  1. POST credentials to /api/auth/login with username and password.
  2. Receive a JWT with typ: "admin" and role: "admin".
  3. Send it as Authorization: Bearer <token> on every admin request.
  4. Expires in 24h. Log in again to rotate.

License Token Usage (Client)

The client gets access_token + refresh_token back from POST /api/licenses/activate. Each access token claims:

json
{
  "sub": 2,                           // user id
  "license_id": 1,                    // entitlement id
  "device_id": 10,                    // activation row id
  "product_code": "tyllink_terminal",
  "typ": "access",                     // "access" | "refresh"
  "iat": 1776576891,
  "exp": 1776577791
}

Device-binding anti-replay: every license endpoint requires device_fingerprint_hash in the body and compares it to the activation's stored value. A stolen token can't be replayed from a different machine — mismatch returns DEVICE_MISMATCH.

Error Responses (admin middleware)

StatusBodyCause
401{"error": "Missing or invalid authorization header"}No Authorization header or missing Bearer prefix
401{"error": "Invalid or expired token"}Token is malformed, tampered, expired, or lacks typ: "admin" (including when a license access token is presented to an admin endpoint)

Rate limits

  • Public endpoints (/api/licenses/*, /api/auth/login): publicLimiter — see Rate Limiting.
  • Sign-in throttle: 5 failed activations per email within a 15-minute sliding window → TOO_MANY_ATTEMPTS. Counter is in-memory; resets on server restart.
  • Admin endpoints: adminLimiter — looser limit because admins make bursts of dashboard calls.

API Reference — License (Public)

The five client-facing endpoints live under /api/licenses/*. /api/license/* (singular) is still mounted as an alias for older clients. Success responses share a common envelope; see the Activate shape. Failures keep the same fields with license_status: "INVALID" and a non-OK reason_code — see Reason Codes.

Authenticates the user, validates the entitlement, records (or refreshes) an activation, and returns access + refresh tokens. This is the only endpoint that accepts raw credentials — every other endpoint uses the access token.

POST /api/licenses/validate Re-check entitlement with the access token

Confirms the current access token is still valid and the entitlement hasn't been revoked/expired server-side. Bumps last_seen_at on the activation. Clients typically call this on app startup after /refresh succeeds.

Headers

Authorization: Bearer <access_token>

Request Body

FieldTypeRequiredDescription
device_fingerprint_hashstringYesMust match the activation's stored value (anti-replay)

Response

Full envelope (same fields as Activate), without the token triple — access_token / refresh_token / expiries are null since no rotation happened.

Reason codes this endpoint returns

OK, SIGNATURE_INVALID, SEAT_REVOKED, LICENSE_EXPIRED, PRODUCT_NOT_ENTITLED, DEVICE_MISMATCH.

cURL Example

bash
curl -X POST http://localhost:4000/api/licenses/validate \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -d '{ "device_fingerprint_hash": "3c05f606..." }'
POST /api/licenses/heartbeat Low-frequency keepalive with optional token rotation

Slim envelope — {status, reason_code} only, plus a rotated token triple if the client sent a refresh_token because the access token had expired. Designed to be cheap enough to call every 4 hours.

Headers

Authorization: Bearer <access_token> — or put the same value in the body field authorization.

Request Body

FieldTypeRequiredDescription
device_idstringYesCross-checked against the token's device_id claim
license_idstringYesCross-checked against the token's license_id claim
authorizationstringNoAlternative to the Authorization header — "Bearer <access>"
refresh_tokenstringNoWhen access is expired, include this to rotate tokens inline

Success Response (valid, no rotation)

json
{ "status": "VALID", "reason_code": null }

Success Response (valid, with rotation)

json
{
  "status": "VALID",
  "reason_code": null,
  "access_token":  "eyJ...",
  "refresh_token": "eyJ...",
  "access_expires_in_sec": 900,
  "refresh_expires_in_sec": 604800
}

Failure Response

json
{ "status": "INVALID", "reason_code": "SEAT_REVOKED" }

Heartbeat's failure reason codes are a narrow set: SEAT_REVOKED, LICENSE_EXPIRED, DEVICE_MISMATCH.

cURL Example

bash
curl -X POST http://localhost:4000/api/licenses/heartbeat \

POST /api/licenses/deactivate Unbind this device (frees a seat)

Marks the activation row as deactivated. Any future call from this machine with the old token returns DEVICE_MISMATCH. Counts back against max_devices, so another device can now take the seat.

Headers

Authorization: Bearer <access_token>

Request Body

FieldTypeRequiredDescription
device_fingerprint_hashstringYesMust match the activation's stored value

Response

Full envelope with license_status: "INVALID" and reason_code: "OK" — semantically "you're signed out; this isn't an error". Tokens are null.

cURL Example

bash
curl -X POST http://localhost:4000/api/licenses/deactivate \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer $ACCESS_TOKEN" \

POST /api/licenses/refresh Trade the refresh token for a new access+refresh pair

Verifies the refresh token (must carry typ: "refresh"), re-checks entitlement and activation status, then mints fresh tokens. Called on app startup from a cached refresh token, and pre-emptively ~2 minutes before access expiry.

Request Body

FieldTypeRequiredDescription
refresh_tokenstringYesJWT with typ: "refresh". Can also arrive via Authorization: Bearer <token>
device_fingerprint_hashstringYesRequired for anti-replay

Response

Full envelope (same as Activate). access_token, refresh_token, and the two *_expires_in_sec fields are populated.

Reason codes this endpoint returns

OK, INVALID_REQUEST, SIGNATURE_INVALID, SEAT_REVOKED, PRODUCT_NOT_ENTITLED, LICENSE_EXPIRED, DEVICE_MISMATCH.

cURL Example

bash
curl -X POST http://localhost:4000/api/licenses/refresh \
  -H "Content-Type: application/json" \
  -d '{
    "refresh_token": "eyJhbGciOi...",
    "device_fingerprint_hash": "3c05f606..."
  }'
GET /api/licenses/devices Self-service: list the signed-in user's own devices

Returns every activation row belonging to the access-token holder across all products, so an end user can see their own seat usage in the app's License → My Devices panel without admin help. Solves the DEVICE_ALREADY_BOUND case: when one machine is already tied to account A and the user wants to sign in as account B on the same machine, they can sign into A, open the portal, and free the seat themselves.

Headers

Authorization: Bearer <access_token> — any access token (typ: access) scoped to a user. No fingerprint check, so this works even when the caller is currently bound to a different device than the one being managed.

Response (200)

      "product_code": "tlink-dev",
      "license_id": 7,
      "device_fingerprint_hash": "3c05f606...",

      "os_version": "25.4.0",

POST /api/licenses/devices/:actId/deactivate Self-service: deactivate one of the user's own devices

Marks the named activation row as deactivated_at = now(). Returns 404 if the activation doesn't belong to the caller. Differs from POST /api/licenses/deactivate in two ways: (1) the activation to free is selected by :actId path param, not the caller's fingerprint, so the user can evict a different machine they're signed into; (2) it accepts any valid access token without a fingerprint header.

Effect on the target device: the next /heartbeat or /refresh from that device returns license_status: "INVALID", reason_code: "SEAT_REVOKED". The client drops to unauthenticated and re-prompts sign-in.

Effect on the caller: if the caller is deactivating their own current activation, the client should not follow up with POST /api/licenses/deactivate — that endpoint requires a valid token matching the now-revoked fingerprint and will 403. Instead, the client clears local tokens directly (deactivateLicense(skipServerCall = true) in the reference client).

Device Fingerprint

The client is responsible for generating a stable per-machine identifier. The reference client (in tlink-license-client/src/lib/services/fingerprint.service.ts) combines five inputs with | separators and hashes them with SHA-256:

text
SHA256(
  machine-uuid  |     // macOS: ioreg IOPlatformUUID
                      // Linux: /etc/machine-id
                      // Windows: HKLM\SOFTWARE\Microsoft\Cryptography\MachineGuid
  primary-mac   |     // first non-loopback interface MAC
  hostname      |     // os.hostname()
  platform      |     // os.platform() e.g. "darwin"
  arch                // os.arch() e.g. "arm64"
)

The resulting 64-character hex string is sent as device_fingerprint_hash. The server stores it verbatim on the activation row and only ever compares equality — the raw inputs never leave the machine.

What changes the fingerprint (orphaning the activation)

  • OS reinstall that regenerates the machine UUID
  • Swapping the primary network adapter or enabling MAC randomization on WiFi
  • Renaming the machine in system preferences
  • Running in a fresh VM

When the hash changes, /activate treats the machine as a new device — counts against max_devices. The user can free the old seat themselves from Settings → License → My Devices inside the app, or an admin can do it via the Users tab → Active Devices → Deactivate.

ⓘ
Fingerprints are a device-identifier, not a device-attestation. Anyone who copies the machine's UUIDs and MAC into a forged environment can mint the same hash. Production deployments should pair fingerprinting with HTTPS pinning and server-side rate limiting.

function fingerprint (): string {
  const mac = /* first non-loopback MAC from os.networkInterfaces() */
  const machineUuid = /* macOS: ioreg / Linux: /etc/machine-id / Windows: registry */

Step 5: Bootstrap on every subsequent launch

Load the refresh token from keychain and trade it for a fresh pair. If the server is unreachable, fall back to cached state for up to the grace window (default 48h).

typescript
async function bootstrap () {
  const bundle = await loadFromKeychain()
  if (!bundle) return promptSignIn()

  const r = await fetch('http://localhost:4000/api/licenses/refresh', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      refresh_token: bundle.refreshToken,
      device_fingerprint_hash: fingerprint(),
    }),
  }).then(r => r.json()).catch(() => null)

  if (r?.license_status === 'VALID') {
    await saveToKeychain({ ...r, lastServerContactAt: Date.now() })
    return 'active'
  }

  // Offline grace
  const ageHours = (Date.now() - bundle.lastServerContactAt) / 3_600_000
  if (ageHours < 48) return 'active-offline'

  await clearKeychain()
  promptSignIn()
}

Step 6: Schedule refresh + heartbeat

  • Refresh timer: setTimeout for (access_expires_in_sec - 120) * 1000 ms. On fire, call /refresh, replace tokens, reschedule.
  • Heartbeat timer: setInterval every 4 hours. Send { device_id, license_id } with Authorization: Bearer <access>. On access-expired, include refresh_token in the body so heartbeat rotates inline.
  • Any INVALID response (SEAT_REVOKED / LICENSE_EXPIRED / DEVICE_MISMATCH) → clear keychain → re-prompt sign-in.

Diagnose by reason_code. The table in Reason Codes lists every value the server emits; the expanded notes below cover the most common failures and how to unblock them.

INVALID_CREDENTIALS — 401
  • Typo in email or password. Emails are normalized (lowercased + trimmed) so case doesn't matter.
  • User was deleted. Admin → Users tab to re-create.
  • Account exists but the password was recently rotated — old clients still use the cached bundle until it expires.
PRODUCT_NOT_ENTITLED — 403
  • User has no entitlement for the product_code sent. Admin → Users tab → expand user → + Add entitlement.
  • Client is sending the wrong product_code (e.g. tyllink-terminal instead of tyllink_terminal).
  • Product not registered in /api/apps. Add it from the Apps tab first.
DEVICE_LIMIT_REACHED — 403

Every seat on the entitlement's max_devices is occupied by a different fingerprint.

  • Option A: Admin → user → Edit entitlement → bump max_devices.
  • Option B: Admin → Active Devices → Deactivate a stale device. The old client returns DEVICE_MISMATCH on its next call and frees the seat.
  • Option C: From the old machine, call POST /api/licenses/deactivate.
DEVICE_ALREADY_BOUND — 403

This fingerprint is already bound to a different user for the same product (e.g. shared machine, account swap). Deactivate the other user's activation from the admin dashboard first, or issue a new fingerprint.

DEVICE_MISMATCH — 403
  • Body device_fingerprint_hash doesn't match the activation's stored value — token possibly stolen and replayed from another machine.
  • Fingerprint changed because of OS reinstall / MAC randomization / hostname change. Re-activate and free the old seat.
  • Activation was force-deactivated by admin. Client needs to re-sign-in.
  • On heartbeat, expired access without a refresh_token supplied also maps here.
APP_VERSION_BLOCKED — 403

Client's app_version is in the product's blocked_app_versions. Upgrade the client or remove the version from the list in Apps → Edit.

TOO_MANY_ATTEMPTS — 429

5+ failed sign-ins per email within 15 minutes. Counter is in-memory per email; wait 15 minutes or restart the server to reset.

SIGNATURE_INVALID — 401
  • Access token expired — call /refresh or let the client's auto-refresh timer rotate.
  • Token signed with a different JWT_SECRET than the current server. Happens after changing the secret without invalidating clients.
  • Admin middleware rejects license tokens (missing typ: "admin").
  • Bearer header missing or malformed.
CORS errors in browser / Electron
  • Add your client origin to CORS_ORIGINS in .env.
  • Electron renderer origins are typically app:// or file://.
  • Set CORS_ORIGINS=* to confirm it's a CORS problem, then narrow to specific origins.
Server unreachable / ECONNREFUSED
  • curl http://localhost:4000/api/health — if this fails, the server isn't running.
  • Check the port matches the client's serverUrl.
  • Firewall / security group rules on port 4000 (or your HTTPS port).
  • Docker: docker logs tlink-license. PM2: pm2 logs tlink-license.
Client stuck in "offline grace" banner

The reference client keeps the last VALID state usable for 48h without reaching the server. If you're stuck past that, check the client's console for the failing /refresh error — usually it's either a server-unreachable problem or a SEAT_REVOKED that the client can't act on while offline.

License Revocation & Deactivation

There are four distinct ways a user can lose access, each with a different scope, reversibility, and admin workflow. Use this table to pick the right one.

Action Scope Reversible? Reason code Typical use
Revoke seat All devices on one entitlement Yes — flip status back to active SEAT_REVOKED Refund, abuse, policy violation, offboarded employee
Delete entitlement One product on one user No — re-add if needed PRODUCT_NOT_ENTITLED User dropped a product from their plan
Force-deactivate device One device only Yes — user can re-activate DEVICE_MISMATCH Lost/stolen laptop, device swap, seat reclaim
Self-service deactivate One device only Yes — user can re-activate SEAT_REVOKED User frees their own seat from inside the app (no admin needed)
Expire All devices on one entitlement No — natural end LICENSE_EXPIRED expiry_date passes

Timeline after any of the above

Admin acts → Next /heartbeat / /validate / /refresh (<= 4h) → Server returns INVALID + reason_code → Client clears keychain, shows sign-in

Enforcement is not instant — it waits for the next client call. With default timers (4h heartbeat, ~13-min refresh, validate on launch), the worst case is about 4 hours. Shorten the heartbeat interval on the client side for tighter control.

Revoke a seat

Best for "this user shouldn't have access" while keeping history.

bash
# Admin token
TOKEN=$(curl -s -X POST http://localhost:4000/api/auth/login \
  -H "Content-Type: application/json" \

Force-deactivate a device

Best when you need to free a seat under max_devices without touching the entitlement. Admin Users tab → expand user → Active Devices → Deactivate, or:

curl -X POST http://localhost:4000/api/users/2/activations/10/deactivate \

Self-service portal (user-initiated, no admin)

Inside the app, Settings → License → My Devices shows every activation row the signed-in user owns across all products, with a THIS DEVICE badge on the current one and a Deactivate button on each row. Solves the DEVICE_ALREADY_BOUND case: user previously signed in on machine A with yahoo.com, now wants to sign in with gmail.com on the same machine — they sign into the yahoo account, open the portal, deactivate that device, sign out, sign in with gmail. No admin interaction.

Backed by two endpoints:

  • GET /api/licenses/devices — list the caller's own activations
  • POST /api/licenses/devices/:actId/deactivate — free one of the caller's seats

Unlike POST /api/licenses/deactivate, these endpoints don't require a fingerprint match — any valid access token scoped to the user works. This lets the user evict a different machine than the one they're calling from.

bash
# List my devices
curl http://localhost:4000/api/licenses/devices \
  -H "Authorization: Bearer $ACCESS_TOKEN"

# Deactivate activation id 42
curl -X POST http://localhost:4000/api/licenses/devices/42/deactivate \
  -H "Authorization: Bearer $ACCESS_TOKEN"

When the caller deactivates their own current device via this portal, the client must not follow up with POST /api/licenses/deactivate — the token is already invalid for the device-scoped endpoint and that call will 403. Instead, clear local tokens directly.

User self-signout (client-initiated)

From the client's perspective this is the clean "sign out of this device" flow. It requires the active session's Bearer token:

bash
curl -X POST http://localhost:4000/api/licenses/deactivate \
  -H "Authorization: Bearer $ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{ "device_fingerprint_hash": "3c05f606..." }'

Offline user behaviour

⚠
Clients running within their 48h offline grace window won't learn about a revocation until they reconnect. Their cached VALID state is honoured until the grace period expires or a server call succeeds. Plan accordingly — for true instant kill, shorten the grace window or push shorter heartbeat intervals.

Offline Activation Codes

For users whose machines can't reach the license server — corporate firewall, air-gapped network, deep-VPN-only policy — admins can mint a signed RS256 JWT that the client verifies purely locally against a public key baked into the app at build time. No server round-trip is required at redemption, so the user gets a working license even when the server is unreachable.

ⓘ
device_fingerprint_hash is required on every mint — an unbound code would let one JWT unlock unlimited machines, and without a fingerprint there's no way for the server to count seats. Minting also pre-creates an activations row against max_devices, so an offline-issued seat is tracked and revocable identically to an online one.

Workflow

  1. User copies their fingerprint — opens the app, goes to Settings → License → Device fingerprint, clicks Copy. The value is a SHA-256 hex digest of machine UUID + primary MAC + hostname + platform + arch, computed locally.
  2. User sends it to admin via whatever out-of-band channel works (email, chat, ticket).
  3. Admin mints the code — /admin → Users tab → expand user → entitlement row → Offline code. Paste the fingerprint, set Valid for (days), click Generate. If the entitlement's max_devices is already saturated, the server rejects with DEVICE_LIMIT_REACHED — deactivate an existing device first.
  4. Admin shares the code — copy the JWT blob and send it back to the user. It's a single string, QR-friendly, email-friendly.
  5. User redeems it — activation dialog → offline mode → paste → Redeem. Client verifies signature (RS256 against baked public key), typ/alg header, exp, and that payload.device_fingerprint_hash matches the local fingerprint. On success, the app unlocks with the claims from the payload.

What lives where after mint

ArtifactLives onPurpose
Signed JWT codeUser's machine (pasted into dialog)Local claims + proof of admin origin (RS256 signature)
activations row (activation_source: "offline", last_seen_at: null)Server DBSeat tracking — counts toward max_devices; appears in admin Active Devices with an offline badge
RSA-2048 private keyServer: ./data/offline-keys/offline-private.pem (mode 0600)Signs codes. Never leaves the server.
RSA-2048 public keyApp bundle: tlink-core/src/offline-public-key.tsVerifies codes offline. Baked in at app build time; shipping a new public key requires a new app release.

Revoking an offline-issued seat

Same path as any other activation: admin goes to Active Devices and clicks Deactivate, or the user self-serves from Settings → License → My Devices. The signed JWT in the user's possession is still cryptographically valid — but at the next time the machine does reach the server (heartbeat, refresh, new activate), that call will return SEAT_REVOKED and the client will drop to unauthenticated. Until then, a fully-offline machine will keep running on the cached code until its exp.

⚠
Offline codes are a one-way trust delegation: you can't remotely kill one until the client comes online. If instant kill is a hard requirement, shorten valid_for_days so a fully-offline machine expires itself quickly, or don't issue offline codes for that user.

  • 400 — missing / malformed fingerprint, valid_for_days out of range, member_user_id not a positive integer, or the named member is not assigned to this entitlement

  • 404 — user, entitlement, or member user not found

Seat accounting

The pre-created activations row consumes one seat from the entitlement's max_devices at mint time. For TEAM entitlements, the count is across the full pool (owner + all members) — matching the online /activate flow — so you can't over-allocate by minting offline codes. Admins can evict an offline-issued seat the same way as an online one (force-deactivate), or the member can self-serve via Settings → License → My Devices.

Idempotency

Re-minting for the same fingerprint on the same entitlement reuses the existing active row — no new seat is consumed, even across different member_user_id values (if a laptop changes hands between team members, the new subject is written to the existing row). The new JWT supersedes the old one; seat count stays the same.

cURL Example

bash
curl -X POST http://localhost:4000/api/users/2/entitlements/1/offline-code \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "device_fingerprint_hash": "3c05f606abcd...",
    "valid_for_days": 30

Client-side verification

The reference client in tlink-license-client validates in this order — any failure returns a user-visible error without changing session state:

  1. JWT shape: three base64url segments, starts with eyJ.
  2. Header pin: alg === "RS256" and typ === "JWT". Rejects alg: none outright.
  3. Signature: crypto.createVerify('RSA-SHA256') against the baked-in public key.
  4. Payload: typ === "offline", exp not in the past.
  5. Fingerprint: payload.device_fingerprint_hash === await this.fingerprint.get() — rejects if the code was minted for a different machine.

Team Membership

A TEAM entitlement's max_devices is a shared seat pool, not a per-user cap. The team owner holds the entitlement; additional users (team members) can be pointed at it and activate using their own email + password. Every member's activation counts against the same pool; a 10-seat team can be 10 different people on 1 laptop each, or 1 person on 10 laptops, or any mix in between.

Data model

Team membership is represented by a single pointer on the member's user record:

user record (member)
{
  "id": 12,
  "email": "alice@team.example.com",
  "name": "Alice",
  "password_hash": "$2b$10$...",
  "entitlements": [],         // members don't own entitlements
  "member_of_license_id": 7  // points at a TEAM entitlement on another user
}

The referenced entitlement is looked up across all users by id; since entitlement ids are globally unique (counter-backed), the lookup is unambiguous. A member can point at exactly one team entitlement at a time.

Activation flow for a member

  1. Member signs in with their own email + password (not the owner's).
  2. Server resolves the entitlement for product_code — first checks the member's own entitlements[] (empty), then falls back to dereferencing member_of_license_id and confirming the product matches and license_type === "TEAM".
  3. Seat-pool check: active_activations WHERE license_id = <team_ent_id> AND deactivated_at IS NULL < max_devices. Pool is shared across owner + all members.
  4. If pool has space, insert an activations row with user_id = the member's id (so devices stay attributable to the person), license_id = the team entitlement id.
  5. Issue access + refresh tokens with sub = member id and license_id = team entitlement id. Heartbeat / refresh resolve the team entitlement the same way.

Provisioning (admin-only)

There's no self-service invite flow in this release — admins add and remove members via /admin:

  1. Create the team owner as usual. Add a TEAM entitlement to them with the target max_devices.
  2. Create the team member: Users tab → + Create User, fill in email + password, and set Team membership to the owner's team entitlement in the dropdown.
  3. To promote or reassign: edit the member and change (or clear) the dropdown. Clearing revokes access on the next heartbeat/refresh — they get PRODUCT_NOT_ENTITLED.
  4. Under the team owner's expanded row, an inline Team Members table lists every user pointing at the owner's TEAM entitlement(s), with a Remove from team button.

Device tracking

Each activation row stores the member's user_id, so the admin can see which device belongs to which member in their Active Devices view. The team owner's Active Devices table shows only the owner's own devices; each member's devices show under the member's row. The team entitlement's Devices column (e.g. 3/10) reflects the pooled count across everyone.

Lifecycle

  • Owner deletes the TEAM entitlement — all activations on it are deleted, and every member's member_of_license_id is cleared. Members drop to unauthenticated on next heartbeat.
  • Owner user is deleted — same cleanup applies to every entitlement they owned. Members' pointers to those entitlements are cleared.
  • Member's member_of_license_id is cleared — next heartbeat/refresh returns PRODUCT_NOT_ENTITLED, client drops to sign-in.
  • Offline codes — minted against a specific entitlement id. An admin can mint an offline code bound to the team entitlement for a member's device; the resulting activations row carries the member's user id and counts against the shared pool just like an online activation.

API surface

  • POST /api/users accepts an optional member_of_license_id. Server validates the id exists and names a TEAM entitlement.
  • PUT /api/users/:id accepts member_of_license_id: pass a number to link, null to unlink. Same validation.
  • GET /api/users/:id responses include team_membership: { license_id, product_code, owner_email, owner_name } when the user is a member (or { license_id, dangling: true } if the pointer is stale). Team entitlements also include a member_count in their serialization.
ⓘ
This is the minimal team model. There's no invite-by-email flow, no self-service "add member" for team owners, and no first-class teams table — just a pointer. If you outgrow this (e.g. need multiple entitlements per team, transferable ownership, or member-initiated joins), the migration path is to promote team_id to its own table and treat this pointer as a compatibility shim.

Client side — which entitlement gets bound

The product_code in the activate body selects the entitlement. The issued tokens carry that product_code + license_id, so a single process is bound to exactly one product. To use both products in parallel, you need two separate client sessions (or launch the second product in its own process).

json
{
  "email": "team-member@example.com",
  "password": "...",
  "product_code": "tyllink_terminal",   // ← picks which entitlement
  "device_fingerprint_hash": "..."
}

Cross-product isolation

An entitlement is scoped to its product_code. Requests that claim the wrong product get PRODUCT_NOT_ENTITLED:

  • A user with only tyllink_terminal cannot activate tyllink_studio.
  • max_devices counts per entitlement, independently for each product.
  • blocked_app_versions is per-app — blocking 0.9.0 of Terminal doesn't affect Studio.
  • Revoking one entitlement leaves the other active.

Bulk provisioning

For batch onboarding (reseller, enterprise order), script account creation + entitlement grants:

bash
# Seed 10 team users with Terminal + Studio entitlements.
for i in $(seq 1 10); do
  # Create user
  USER=$(curl -s -X POST http://localhost:4000/api/users \
    -H "Authorization: Bearer $TOKEN" \

    -d "{\"email\":\"user$i@acme.com\",\"password\":\"temp-pw\",\"name\":\"User $i\"}")

  # Grant both products
  for product in tyllink_terminal tyllink_studio; do
    curl -s -X POST "http://localhost:4000/api/users/$USER_ID/entitlements" \

      -d "{\"product_code\":\"$product\",\"license_type\":\"TEAM\",\"billing_type\":\"PAID\",\"expiry_date\":\"2027-12-31\",\"max_devices\":3}"
  done
done
ⓘ
For large batches, pace requests to avoid TOO_MANY_ATTEMPTS on any subsequent sign-in attempts (5 failed per email / 15-minute sliding window). Admin endpoints are not rate-limited on that logic — only sign-in through /activate is.

Process Management & Auto-Restart

The Problem

Running the license server directly with node src/index.js has significant limitations:

  • Using "Restart Server" from the admin dashboard stops the server permanently — there is no parent process to restart it.
  • If the server crashes, it stays down until manually restarted.
  • The server does not automatically start after a system reboot.
⚠
Do not use node src/index.js directly in production. Use one of the process management solutions below to ensure the server automatically restarts after crashes, reboots, and dashboard-initiated restarts.
