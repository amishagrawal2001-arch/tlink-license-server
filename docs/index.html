Tlink License Server

A centralized license management server for Tlink products. Generate, activate, validate, and manage license keys for your Electron/desktop applications with a full-featured admin dashboard and REST API.

ⓘ
Base URL: http://localhost:4000 (development)
All API endpoints are prefixed with /api.

Quick Start

  1. Clone and install dependencies
    bash
    git clone <repo-url> tlink-license-server
cd tlink-license-server
npm install
  2. Configure environment
    bash
    cp .env.example .env
# Edit .env with your settings (see Configuration section)
  3. Seed the database
    bash
    npm run seed

    This creates an admin user, registers the NetOps app (NO), and generates sample license keys.

  4. Start the server
    bash
    # Development (with auto-reload)
npm run dev

# Production
npm start
  5. Open the admin dashboard

    Navigate to http://localhost:4000/admin and login with admin / admin123.

How It Works

The license system follows a straightforward flow from download through ongoing validation. Below are the key workflows visualized end-to-end.

End-to-End License Lifecycle

Trial & Activation Flow

Customer Downloads App → 14-Day Trial Starts → Trial Expires → Must Activate License

Key Generation Flow

Admin Opens Dashboard → Generates License Key → Sends Key to Customer (email, Slack, etc.)

Activation Flow

Customer Enters Key → App Calls /api/license/activate → Server Validates Key → Returns tier, expiry

Ongoing Heartbeat (every 4 hours)

App Sends Heartbeat → Server Checks Key Status → Returns { valid: true/false }

Revocation Flow

Admin Revokes Key → Status Set to "revoked" → Next Heartbeat Fails → App Shows Expired Dialog

Installation Guide

Multiple installation methods are available. Choose the one that fits your infrastructure.

Install from GitHub Releases (Recommended)

Download the latest release archive for your platform from the GitHub Releases page.

macOS

bash
# Download the release
curl -LO https://github.com/tlink/license-server/releases/latest/download/tlink-license-server-macos.tar.gz

# Extract
tar xzf tlink-license-server-macos.tar.gz
cd tlink-license-server

# Run the installer
sudo bash install-mac.sh

The installer creates a LaunchDaemon at /Library/LaunchDaemons/io.tlink.license-server.plist and starts the service automatically.

Database Setup

The server uses a flat-file JSON database stored at the configured DATABASE_PATH (default: data/licenses.json). No external database is required.

The database is automatically created on first run. It contains four collections:

CollectionPurpose
appsRegistered applications (app_code, app_name)
license_keysGenerated license keys with metadata
activationsMachine activations linked to keys
admin_usersAdmin users with bcrypt-hashed passwords
ⓘ
Run npm run seed after initial setup to populate the database with an admin user, the NetOps app, and sample license keys.

CORS Configuration

Set the CORS_ORIGINS environment variable to control which origins can make requests to the API.

  • * — Allow all origins (development only)
  • Comma-separated list — e.g., http://localhost:4200,app://tlink-netops

For Electron apps, use the app:// protocol scheme as your origin.

Rate Limiting

Two rate limit tiers are applied via express-rate-limit:

TierRoutesWindowMax Requests
Public/api/auth/*, /api/license/*60 seconds30 requests
Admin/api/keys/*, /api/dashboard/*60 seconds100 requests

Rate limit headers are returned in responses (RateLimit-Limit, RateLimit-Remaining, RateLimit-Reset).

Authentication

Admin Login Flow

  1. POST credentials to /api/auth/login with username and password.
  2. Receive a JWT token in the response along with the user object.
  3. Include the token in subsequent requests as Authorization: Bearer <token>.
  4. Token expires in 24 hours. Re-authenticate to obtain a fresh token.
bash
# Login and get token
curl -X POST http://localhost:4000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username": "admin", "password": "admin123"}'
json
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "user": {
    "id": 1,
    "username": "admin",
    "role": "admin"
  }
}

JWT Token Usage

The JWT payload includes id, username, and role. Tokens are signed with the JWT_SECRET and expire after 24 hours.

Include the token in the Authorization header for all admin endpoints:

bash
curl http://localhost:4000/api/keys \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..."

Key Format

Tlink license keys follow a structured format that encodes application, tier, and expiry information with a checksum for integrity verification.

Structure

TLINK-NO-P4KR-2712-X9HJ-AB3F | | | | | | | | | | | +-- Checksum (FNV-1a hash, 4 chars) | | | | +---------- Random segment (4 chars) | | | +------------------ Expiry: YY=27, MM=12 (Dec 2027) | | +-------------------------- Tier char + 3 random (P=Pro, E=Enterprise) | +-------------------------------- App code (2 chars, padded with X) +---------------------------------------- Prefix (always TLINK)

Segment Details

SegmentLengthDescription
TLINK5Fixed prefix identifying Tlink keys
App Code2Two-letter application code, padded with X if needed
Tier Segment4First character encodes tier (P=Pro, E=Enterprise), followed by 3 random characters
Expiry4Two-digit year + two-digit month (YYMM)
Random4Random characters from charset A-Z (no I, O) + 2-9
Checksum4FNV-1a hash of all preceding segments + salt, encoded as 4 characters

Checksum Algorithm

The checksum is computed using FNV-1a (Fowler-Noll-Vo) hash:

  1. Concatenate all segments with dashes: TLINK-NO-P4KR-2712-X9HJ
  2. Append the KEY_SALT environment variable: TLINK-NO-P4KR-2712-X9HJ-<salt>
  3. Compute FNV-1a hash (32-bit)
  4. Convert to 4-character string using the charset: ABCDEFGHJKLMNPQRSTUVWXYZ23456789
ⓘ
The charset intentionally excludes I, O, 0, and 1 to avoid visual ambiguity when users type keys manually.

Tier Encoding

CharacterTierDescription
PProProfessional tier
EEnterpriseEnterprise tier

Client Integration Guide

Complete guide for integrating any application (Electron, web, desktop) with the Tlink License Server. Follow these five steps to add full license management to your app.

Step 1: Register Your App

Before generating keys, register your application with the license server. Each app gets a unique 2-letter code.

bash
# First, get an admin token
TOKEN=$(curl -s -X POST https://your-server:4000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"username":"admin","password":"your-password"}' | jq -r .token)

# Register your application
curl -X POST https://your-server:4000/api/keys/apps/register \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "appCode": "MA",
    "appName": "My Awesome App"
  }'

# Response: { "id": 2, "appCode": "MA", "appName": "My Awesome App" }
ⓘ
App codes are uppercase, 2 characters max. They are embedded in every license key (e.g., TLINK-MA-...) so the server can verify a key belongs to the correct application.

Step 2: Add License Service to Your App

Create a LicenseService class that handles all license operations. This example works in TypeScript/JavaScript for Electron or browser apps.

typescript
// license.service.ts - Complete License Service

interface LicenseData {
  key: string;
  tier: 'pro' | 'enterprise';
  expiry: string;
  customer: string;
  activatedAt: string;
  lastValidatedAt: string;
}

interface LicenseResponse {
  valid: boolean;
  tier?: 'pro' | 'enterprise';
  expiry?: string;
  customer?: string;
  message?: string;
  reason?: string;
  daysRemaining?: number;
}

export class LicenseService {
  private SERVER_URL = 'https://license.yourdomain.com';
  private APP_CODE = 'MA';           // Your registered app code
  private APP_VERSION = '1.0.0';
  private GRACE_DAYS = 7;
  private HEARTBEAT_MS = 4 * 60 * 60 * 1000;  // 4 hours
  private heartbeatTimer: any = null;

  // ── Activate a license key ──
  async activate(key: string): Promise<LicenseResponse> {
    const res = await fetch(`${this.SERVER_URL}/api/license/activate`, {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        key,
        machineId: this.getMachineId(),
        appCode: this.APP_CODE,
        appVersion: this.APP_VERSION,
        os: this.getOS()
      })
    });

    const data: LicenseResponse = await res.json();

    if (data.valid) {
      this.saveLicense({
        key,
        tier: data.tier!,
        expiry: data.expiry!,
        customer: data.customer || '',
        activatedAt: new Date().toISOString(),
        lastValidatedAt: new Date().toISOString()
      });
      this.startHeartbeat();
    }

    return data;
  }

  // ── Validate on startup ──
  async validate(): Promise<LicenseResponse> {
    const saved = this.getSavedLicense();
    if (!saved) return { valid: false, reason: 'No license found' };

    try {
      const res = await fetch(`${this.SERVER_URL}/api/license/validate`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          key: saved.key,
          machineId: this.getMachineId()
        })
      });

      const data: LicenseResponse = await res.json();

      if (data.valid) {
        saved.lastValidatedAt = new Date().toISOString();
        saved.tier = data.tier!;
        saved.expiry = data.expiry!;
        this.saveLicense(saved);
        this.startHeartbeat();
      }

      return data;
    } catch (err) {
      // Offline fallback
      return this.checkOffline(saved);
    }
  }

  // ── Heartbeat (called every 4 hours) ──
  async heartbeat(): Promise<boolean> {
    const saved = this.getSavedLicense();
    if (!saved) return false;

    try {
      const res = await fetch(`${this.SERVER_URL}/api/license/heartbeat`, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify({
          key: saved.key,
          machineId: this.getMachineId()
        })
      });

      const data = await res.json();

      if (data.valid) {
        saved.lastValidatedAt = new Date().toISOString();
        this.saveLicense(saved);
      }

      return data.valid;
    } catch (err) {
      console.warn('Heartbeat failed, will retry next interval');
      return true; // Allow during grace period
    }
  }

  // ── Offline fallback logic ──
  private checkOffline(saved: LicenseData): LicenseResponse {
    const daysSince = (Date.now() - new Date(saved.lastValidatedAt).getTime())
      / (1000 * 60 * 60 * 24);
    const notExpired = new Date() < new Date(saved.expiry);
    const inGrace = daysSince < this.GRACE_DAYS;

    if (notExpired && inGrace) {
      return { valid: true, tier: saved.tier, expiry: saved.expiry };
    }
    return { valid: false, reason: 'Offline grace period expired' };
  }

  // ── Persistence helpers ──
  private saveLicense(data: LicenseData) {
    localStorage.setItem('tlink_license', JSON.stringify(data));
  }

  getSavedLicense(): LicenseData | null {
    const raw = localStorage.getItem('tlink_license');
    return raw ? JSON.parse(raw) : null;
  }

  private startHeartbeat() {
    if (this.heartbeatTimer) clearInterval(this.heartbeatTimer);
    this.heartbeatTimer = setInterval(() => this.heartbeat(), this.HEARTBEAT_MS);
  }

  // ── Platform helpers ──
  private getMachineId(): string {
    // Generate or retrieve a persistent machine identifier
    let id = localStorage.getItem('tlink_machine_id');
    if (!id) {
      id = crypto.randomUUID();
      localStorage.setItem('tlink_machine_id', id);
    }
    return id;
  }

  private getOS(): string {
    if (typeof process !== 'undefined') return process.platform;
    return navigator.platform || 'browser';
  }
}

Step 5: Initialize on App Startup

Wire everything together in your app's initialization code.

typescript
// app-init.ts - Call this when your app starts

const licenseService = new LicenseService();

async function initApp() {
  const TRIAL_DAYS = 14;
  const saved = licenseService.getSavedLicense();

  if (saved) {
    // Has a saved license - validate it
    const result = await licenseService.validate();
    if (result.valid) {
      updateBanner(result.tier!, 0, result.tier);
      enableAllFeatures();
    } else {
      updateBanner('expired', 0, null);
      disableEditing();
    }
  } else {
    // No license - check trial
    const installDate = localStorage.getItem('install_date')
      || (() => { localStorage.setItem('install_date', new Date().toISOString()); return new Date().toISOString(); })();

    const daysUsed = Math.floor((Date.now() - new Date(installDate).getTime()) / (86400000));
    const daysLeft = Math.max(0, TRIAL_DAYS - daysUsed);

    if (daysLeft > 0) {
      updateBanner('trial', daysLeft, null);
      enableAllFeatures();
    } else {
      updateBanner('expired', 0, null);
      disableEditing();
      showActivationDialog();
    }
  }
}

Admin Workflow

Step-by-step guide for administrators managing license keys through the dashboard.

  1. Access the Dashboard

    Navigate to http://your-server:4000/admin in any modern browser. The root URL / automatically redirects to /admin.

  2. Login with Admin Credentials

    Enter your admin username and password. Default credentials are admin / admin123 (change these in production via the ADMIN_USERNAME and ADMIN_PASSWORD environment variables).

  3. Generate Keys for Customers

    From the dashboard, select the target application, choose a tier (Pro or Enterprise), set an expiry date, and optionally enter customer details. Click "Generate Key" and the system will create a formatted key like TLINK-NO-P4KR-2712-X9HJ-AB3F.

  4. Send Key to Customer

    Copy the generated key and send it to your customer via email, Slack, or any secure channel. The key is self-contained and does not need additional setup on the customer side.

  5. Monitor Activations

    The dashboard shows real-time activation data: which machines have activated each key, when they last checked in (heartbeat), their OS, and app version. Use this to verify successful deployments.

  6. Revoke Keys if Needed

    If a customer's subscription ends or a key is compromised, revoke it from the dashboard. The key status changes to "revoked" and all activated machines will fail on their next heartbeat check (within 4 hours).

  7. Manage Settings

    Adjust server configuration via environment variables: change the port, update CORS origins, modify rate limits. Restart the server after changes to .env.

Troubleshooting

400: Invalid license key

The key format or checksum does not match. Common causes:

  • Key was typed incorrectly (check for O vs 0, I vs 1 confusion — these characters are excluded from valid keys)
  • The KEY_SALT on the server differs from when the key was generated. Keys generated with a different salt will not validate.
  • Key string was truncated during copy-paste
403: Max activations reached

The license key has been activated on the maximum number of machines. Solutions:

  • Deactivate a machine using POST /api/license/deactivate
  • Ask the admin to increase maxMachines via PUT /api/keys/:id
401: Missing or invalid authorization header

Admin API calls require the Authorization header:

  • Ensure the header format is exactly: Authorization: Bearer <token> (note the space after "Bearer")
  • Token may have expired (24-hour validity). Re-authenticate via POST /api/auth/login.
CORS errors in browser/Electron

The browser is blocking requests due to cross-origin policy:

  • Check that CORS_ORIGINS includes your app's origin
  • For Electron apps, the origin is typically app://<app-name> or file://
  • Set CORS_ORIGINS=* temporarily to confirm CORS is the issue
  • Ensure no typos in the origin URL (protocol, port, trailing slash)
CSP (Content Security Policy) blocking requests

If your Electron app has CSP enabled, it may block connections to the license server:

  • Add the license server URL to the connect-src directive in your CSP meta tag
  • Example: connect-src 'self' http://localhost:4000 https://license.yourdomain.com
  • Check the browser/Electron dev console for CSP violation messages
Server unreachable / ECONNREFUSED

The license server is not responding:

  • Verify the server is running: curl http://localhost:4000/api/health
  • Check the configured port matches your client URL
  • Ensure firewall rules allow traffic on port 4000
  • In Docker: check container logs with docker logs tlink-license
  • With PM2: check logs with pm2 logs tlink-license
Key validation passes but activation fails

The key structure is valid but the database record check fails:

  • The key may not exist in the database (was generated but never inserted)
  • The key status may be revoked
  • The appCode in the request may not match the key's associated app
Rate limit exceeded (429)

Too many requests from the same IP address. The server enforces:

  • Public routes: 30 requests per minute
  • Admin routes: 100 requests per minute
  • Check RateLimit-Remaining header to monitor usage
  • Implement exponential backoff in your client
